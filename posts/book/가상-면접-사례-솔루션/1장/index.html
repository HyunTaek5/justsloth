1장. 사용자 수에 따른 규모 확장성

단일 서버

웹, 앱, DB, 캐시를 하나의 서버에서 전부 관리하는 방식

  • AWS 기준으로 EC2 인스턴스 하나에 모든 서비스를 올리는 방식이라 생각하면 된다.

데이터베이스

사용자가 많아지면, 서버를 여러대 두면서 용도에 따라 서버를 분리하게된다.

  • 트래픽을 받는 웹서버 / DB 서버 / 캐시 서버 등으로 분리한다.

데이터베이스는 전통적인 SQL 기반 관계형 데이터베이스와 비관계형 데이터베이스로 나뉜다.

  • SQL 기반 데이터베이스는 트랜잭션 처리가 가능하고, 데이터의 일관성을 보장한다.
  • 비관계형 데이터베이스는 데이터의 일관성을 보장하지 않지만, 대용량 데이터를 처리할 수 있다.
    • 비정형 데이터이거나, 직렬화/역직렬화만 가능하면 된다.

수직적 규모 확장 vs 수평적 규모 확장

스케일 업이냐 스케일 아웃이냐

  • 스케일 업(수직적 규모 확장) :
    • CPU, 메모리, 디스크 용량 등 서버 자원을 추가하는 방식
    • 한 대의 서버에 추가할 수 있는 자원이 한계가 있다
    • 장애에 대한 자동복구/다중화 방안을 제시하지 못한다
  • 스케일 아웃(수평적 규모 확장)
    • 더 많은 수의 서버를 추가하여 성능을 개선하는 방식

로드 밸런서

로드 밸런서는 서버의 트래픽 부하를 여러 서버에 고르게 분산하는 역할을 한다. 서버 간 통신을 위해 사설 IP를 사용하고, 로드 밸런서는 공인 IP를 통해 외부에 api를 연결한다.

  • 로드 밸런서는 서버가 다운된 경우, 다른 서버로 트래픽을 전송하여 failover에 대한 전략을 세울 수 있다.
  • 트래픽이 급증할 경우, 로드밸런싱 그룹에 서버를 새로 추가하여 부하를 분산시켜 가용성을 높일 수 있다.

DB 다중화

DB 서버를 여러 대두고 master - slave 관계로 master에서는 쓰기 작업을, slave에서는 읽기 작업을 처리한다.

DB 다중화를 통해 더 나은 DB 성능, 안정성, 가용성을 확보할 수 있다.

  • master가 다운된 경우, slave 중 하나를 master로 승격시켜 가용성을 높일 수 있다.
  • 데이터의 일관성을 위해 master-slave 간 동기화를 유지해야한다.
